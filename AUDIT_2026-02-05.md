# Smart Invoice Workflow — Full Audit

**Date:** 2026-02-05
**Auditors:** Claude (Super Manager), Claude (Cursor/Opus), Gemini (Cursor)
**Verdict:** Working prototype, never deployed. Solid routing logic, significant gaps in test coverage, security, and deployment readiness.

---

## What It Does

An automated invoice collection system for service businesses. The workflow:

1. Reads overdue invoices from a **Google Sheet** (the source of truth)
2. Calculates how many days each invoice is overdue
3. Determines which **escalation stage** to send (6 stages over 42 days)
4. Creates **Gmail drafts** (NOT auto-send) with templated, escalating reminders
5. Writes tracking data back to Google Sheets (last stage sent, date sent)
6. Human reviews drafts in Gmail and clicks Send

**Key design choice:** Drafts only, never auto-sends. Human stays in the loop.


## Issues Found

### Critical — Must Fix Before Any Use

1. **`email_autoresponder.json` contains third-party credentials** — Hardcoded account ID (`2173139`), OpenAI connection ID (`634823`), email addresses (`nick@leftclick.ai`, `nick@1secondcopy.com`), brand name (`1SecondCopy`). This is someone else's Make.com setup committed without sanitization. Sanitize or remove immediately.

2. **Scripts are broken** — `scripts/validate_project.py` imports `scaffold.utils`, `scaffold.alerts`, `scaffold.constants` which don't exist in this project (they're in `project-scaffolding`). Running these scripts crashes with `ModuleNotFoundError`. `pre_review_scan.sh` calls them and also uses `python` directly instead of `uv run`.

3. **Build system broken** — `pyproject.toml` has `build-backend = "setuptools.build_backend"` (should be `setuptools.build_meta`). `uv run pytest` fails.

4. **OAuth flow requires a browser** — Both `sheets.py` and `emailer.py` use `InstalledAppFlow.run_local_server(port=0)`. This opens a browser for auth. **Will not work on any headless server, Docker container, or CI/CD pipeline.** Needs Service Account auth or headless OAuth flow for automated deployment.

5. **Zero test coverage on critical paths** — sheets.py, emailer.py, scheduler.py, config.py, models.py have no meaningful tests.

### High — Bugs and Security Issues

6. **OAuth tokens written world-readable** — `emailer.py:51` and `sheets.py:49` write tokens with default `0o644` permissions. These tokens grant access to Gmail and Sheets. Should use `0o600`.

7. **Settings class evaluated at import time** — All `Settings` attributes are class-level, evaluated when the module is imported. Env var changes after import are invisible. Non-integer `WINDOW_START` crashes at import with unhelpful traceback. `settings.DRY_RUN = True` in `main.py` works by accident (instance attribute shadows class attribute).

8. **Double template rendering** — `scheduler.py:128-146` renders templates once for preview/logging, then `create_draft_from_template()` renders again internally. Wasteful and creates divergence risk if templates had side effects.

9. **`is_overdue()` returns True for paid invoices** — `models.py:56`: `return self.status.lower() == "overdue" or self.days_overdue(today) > 0`. A paid invoice with a past due date reports as overdue. The scheduler filters by status separately (line 92), so no production bug today, but the method is misleading.

10. **Zero-amount invoices pass validation** — `models.py:46` checks `amount < 0` but not `<= 0`. A $0.00 invoice would trigger reminder emails.

11. **Email validation is just `@` check** — `models.py:43`: `"@" not in self.client_email`. Addresses like `"@"`, `"foo@"`, `"@bar"` all pass.

12. **Write-back has race condition** — `sheets.py:write_back_invoices` re-reads the entire spreadsheet to find row indices. If rows shift between initial read and write-back, data goes to wrong rows. O(N*M) complexity.

13. **Write-back initial read has no retry** — `sheets.py:216-222`: The `values().get()` call in `write_back_invoices` doesn't use `_retry_api_call()`, though the subsequent `batchUpdate()` does. A 429 on the read crashes the write-back.

14. **Gmail/Sheets service created per call** — `_get_gmail_service()` and `_get_sheets_service()` build a new service object on every API interaction. No caching. Could trigger OAuth rate limits when processing many invoices.

15. **`DraftCreated.__str__` hardcodes `$`** — `models.py:74`: Always shows `$` regardless of the `currency` field. EUR invoice displays as `$1,500.00`.

16. **No idempotency beyond same-day check** — `router.py:80`: If the Sheet write-back fails after drafts are created, `last_stage_sent` is never updated. Next run re-creates duplicate drafts.

### Medium — Should Fix

17. **Unused dependencies** — APScheduler and rich listed but never imported.

18. **pandas is overkill** — Used solely in `sheets.py` to create a DataFrame from a list of lists and iterate row-by-row. ~30MB dependency for what plain Python dicts could do.

19. **README is stale** — References `utils.py` and `test_sheets.py` which don't exist. Missing `emailer.py`, `scheduler.py`, `templates/`, `test_error_tracking.py`, `test_templates.py`.

20. **No timezone handling** — `config.py:80` uses `datetime.now().hour`, `scheduler.py:66` uses `date.today()`. All naive local time. Docker/cloud defaults to UTC, misaligning business hours.

21. **LOGS_DIR created but never used** — `config.py:37` defines `LOGS_DIR` and `validate()` creates it, but no file handler is ever attached. The `logs/` directory is always empty.

22. **Template regex parsing is brittle** — `emailer.py:77` uses `re.match(r"Subject:\s*(.*?)\n\n(.*)", ...)`. Extra whitespace or newline deviations break parsing.

23. **`sys.path` hacks in every entry point** — `main.py:19` and all test files manually do `sys.path.insert(0, ...)`. Brittle, breaks with `python -m`, conflicts with proper packaging.

24. **Inconsistent error handling** — `emailer.py:155` and `sheets.py:196` wrap `HttpError` in generic `Exception`, losing the original type and traceback. `scheduler.py:88` correctly re-raises.

### Low — Minor Issues

25. **`.gitignore` has HTML comment syntax** — `<!-- SCAFFOLD:START -->` is not valid gitignore syntax. Interpreted as glob patterns.

26. **Template existence test is a no-op** — `test_templates.py:100-101`: `if template_file.exists(): assert template_file.exists()`. Can never fail.

27. **Tests leak temp files on failure** — `test_templates.py` uses `NamedTemporaryFile(delete=False)` with manual `unlink()`. If test fails before cleanup, files leak. Should use pytest `tmp_path` fixture.

28. **Version pinning drift** — `pyproject.toml` uses `>=` (minimum), `requirements.txt` uses `==` (exact). Installing from different sources gives different packages.

29. **Docker runs as root** — No `USER` directive. No `.dockerignore`. No credential mounting strategy. No health check. `CMD` doesn't support passing `--dry-run` without override.

30. **DECISIONS.md is empty** — Only scaffolding boilerplate, no actual decisions recorded.

## What Works Well

- **Clean separation of concerns** — router.py is pure logic, emailer.py handles API, scheduler.py orchestrates
- **Retry logic with exponential backoff** — both Sheets and Gmail have proper 429 handling
- **DRY_RUN mode** — defaults to true, safe by design
- **Template system** — simple `{{variable}}` substitution, easy to customize, professional escalating tone
- **Error tracking** — `ErrorType` enum + `ProcessingError` dataclass gives structured error reporting
- **Draft-only design** — human stays in the loop, can't accidentally blast emails
- **Stage routing logic** — well-tested, handles edge cases (not overdue enough, already sent today, prevent re-sends)
- **Type hints** — consistent throughout the codebase
- **Good `.gitignore` coverage** — credential files, tokens, env files all properly excluded

## Build System Issue

`pyproject.toml` specifies:
```toml
build-backend = "setuptools.build_backend"
```

This is wrong. The correct value is `setuptools.build_meta`. This causes `uv run pytest` to fail. Tests can only be run via `.venv/bin/python -m pytest` as a workaround.

## Summary

Smart-invoice-workflow is a **well-designed prototype** that has never touched production. The routing logic is solid and well-tested. The Google API integration exists but is untested, unconfigured, and has several bugs (double rendering, race conditions, no idempotency). Security needs attention (world-readable tokens, weak email validation, third-party credentials in repo).

**30 issues found across 3 independent audits.** 5 critical, 11 high, 7 medium, 7 low.

**Before this can be useful, we need to decide:** Is this a product Erik wants to use for his own business, sell as a product, or sunset? That decision drives whether we invest in deployment, testing, and productization — or simplify it into something smaller.
